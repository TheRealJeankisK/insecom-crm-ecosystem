Introducción

Contexto y Problemática

El Problema
  • Sistemas Fragmentados: Las aplicaciones del hospital no se comunican de forma fluida, retrasando la visualización de exámenes.
  • Monitoreo Lento: Los signos vitales de pacientes críticos se evalúan manualmente, impidiendo una reacción inmediata ante anomalías.
  • Incompatibilidad de Datos: Laboratorios externos envían exámenes en formatos incompatibles (JSON, XML), causando reprocesos.
¿Por qué SGIP?
  • Integración de Datos: Permite recibir exámenes de laboratorios externos sin alterar el código central (Patrón Adapter).
  • Alertas Clínicas en Vivo: Procesa de forma paralela signos vitales críticos (Fiebre, Hipoxia, Presión) para notificar al médico en segundos.
  • Arquitectura Escalable: Diseñado con microservicios y colas asíncronas para soportar alta demanda sin colapsar el sistema.

Patrones de Diseño

Uso de Broker y Adapter

Patrón Broker
Mensajería asíncrona con RabbitMQ
  • Al crear/actualizar un paciente, la API publica un evento JSON en la cola patients_queue
  • El Lambda Worker consume el evento de forma asíncrona y escribe la alerta en Redis
  • Desacoplamiento total: el backend no espera al worker — responde en ≤15ms
  • Si el worker cae, RabbitMQ persiste los mensajes hasta que se recupere
Patrón Adapter
Integración de Exámenes de Lab
  • Los laboratorios externos envían datos en JSON o XML con formatos distintos
  • El endpoint /ingest-lab actúa como Adapter: normaliza cualquier formato al modelo interno
  • Soporta proveedores: Central JSON, HL7 FHIR, DICOM XML
  • Integración transparente: el sistema no cambia aunque cambie el proveedor externo

Server Functions

Lambda / Serverless Function

  • notification-worker — función Python que corre en su propio contenedor Docker
  • Implementada siguiendo el patrón Azure Functions con function_app.py + host.json
  • El trigger es la llegada de un mensaje a la cola RabbitMQ (event-driven)
  • No expone HTTP — solo consume, procesa y escribe en Redis
  • En producción Azure: se despliega como Azure Function en plan de consumo (pago por uso)
# lambda_runner.py — Event trigger function def start_consuming(): connection = pika.BlockingConnection(...) channel.basic_consume( queue='patients_queue', on_message_callback=process_event ) channel.start_consuming() # blocks — serverless loop
Lo que hace la Lambda
1 · Recibe evento de cola
JSON con datos del paciente creado/actualizado/eliminado
2 · Motor de Reglas Clínicas (CDSS)
Analiza signos vitales con Regex (Temp, FC, SatO2, PA) y dispara alertas críticas.
3 · Escribe en Redis LPUSH
Lista patients_notifications_list con TTL — máx. 50 alertas
4 · Frontend polling
API lee Redis y muestra alertas en tiempo real al médico

Persistencia y Caché

Capas de Datos

MySQL 8 — Datos Transaccionales
Capa de Datos 1 — Docker Volume
  • Tablas: patients, usuarios
  • ORM con SQLAlchemy — modelos tipados
  • Índices B-Tree en campos de búsqueda
  • Volumen persistente en sgip_mysql_data
Redis — Caché en Memoria
Capa de Datos 2 — Docker Volume
  • Lista patients_notifications_list con TTL
  • Latencia de lectura < 2ms
  • Reduce 85% consultas directas a MySQL
Modelo de Mapeo (ORM)

El backend utiliza SQLAlchemy para comunicar el código de la aplicación con la base de datos de forma automática, sin escribir consultas SQL manuales:

Paciente (Tabla pacientes): Almacena nombre, correo, teléfono, diagnóstico principal, tipo de examen solicitado y signos vitales estructurados.
Usuario (Tabla usuarios): Guarda los datos de autenticación del personal médico, encriptando contraseñas con SHA-256 + Salt y gestionando roles (Admin / Doctor).

✓ Ventaja principal: Evita errores de sintaxis y protege la base de datos contra ataques comunes como inyección SQL.

Principios de Diseño

Principios SOLID Aplicados

S — Single Responsibility
Cada módulo tiene una responsabilidad: models.py solo define entidades, queue_manager.py solo publica mensajes, database.py solo genera sesiones
O — Open/Closed
El endpoint Adapter /ingest-lab está abierto a extensión (nuevos proveedores de lab) pero cerrado a modificación del contrato principal de la API
L — Liskov Substitution
Los modelos SQLAlchemy heredan de Base. Cualquier modelo puede sustituirse sin romper las operaciones CRUD del ORM
I — Interface Segregation
Pydantic schemas separados por caso de uso: PatientCreate, PatientUpdate, PatientResponse — sin campos innecesarios en cada contrato
D — Dependency Inversion
FastAPI Depends(get_db) y Depends(verify_token) — los endpoints dependen de abstracciones inyectadas, no de implementaciones concretas. Facilita testing y mocking
